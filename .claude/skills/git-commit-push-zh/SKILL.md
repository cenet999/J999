# 中文提交并推送（git commit + push）

这个 skill 是**手动调用**的执行型工作流。调用后，目标是直接完成一次安全的 `git commit` 和 `git push`，而不是停留在建议层面。

## 适用场景

当用户已经明确要你执行这些动作时使用，例如：

- “帮我提交并推送”
- “git commit 一下再 push”
- “把当前改动推上去”
- “保存到 git 并同步远端”

如果用户只是想了解改了什么、如何写 commit message、是否应该提交，先正常对话，不要直接执行这个 skill。

## 核心原则

1. 先看清楚再提交：先检查工作区和差异，再决定暂存范围。
2. 提交说明使用中文：首行概括改动；必要时补充简短正文说明原因或模块。
3. 默认只提交与当前请求相关的文件，不要顺手把无关改动一起带上。
4. 若没有可提交改动，直接说明并停止，不要制造空提交。
5. 不要擅自使用 `git push --force`、`--no-verify`、`--amend` 等高风险参数。
6. 远端推送属于高影响操作；只有在本 skill 被手动调用、且用户意图明确为提交并推送时才执行 push。

## 执行流程

按下面顺序执行：

1. 运行 `git status --short` 和 `git diff --stat`，必要时再看 `git diff`。
2. 判断哪些文件与用户当前需求相关：
   - 如果用户明确指定了文件或范围，只提交这些文件。
   - 如果用户明确说“当前这些都提交”，再考虑提交全部相关改动。
   - 对 `.env`、密钥、证书、数据库文件、本机配置等敏感或环境专属文件保持警惕，除非用户明确要求且确认安全，否则不要提交。
3. 查看最近提交信息风格，拟定**中文** commit message。
4. 用明确文件路径进行 `git add`，避免无脑 `git add .`；只有在确认当前所有改动都应纳入时才扩大范围。
5. 创建新提交，不要 amend 旧提交。
6. 执行 `git push`：
   - 若当前分支已有上游，直接 push。
   - 若没有上游，可用 `git push -u origin HEAD`。
   - 如果认证失败或远端拒绝，解释原因并停止，不要猜测凭据或乱改配置。
7. 最后汇报：提交哈希、提交标题、push 结果，以及是否还有未提交改动。

## 提交说明写法

提交标题尽量简洁，说明“为什么做这次改动”。

示例：

```text
移除 J9_APP_103 环境变量文件的 Git 跟踪
```

多模块或改动较多时，可写正文，例如：

```text
完善首页图标与数据结构调整

- 更新 pg51 clone 首页展示
- 调整图标定义与类型声明
```

## 遇到异常时的处理

- **工作区为空**：直接告诉用户当前没有可提交内容。
- **存在明显无关改动**：只暂存相关文件，并在汇报里说明还有未纳入的改动。
- **pre-commit hook 失败**：修复问题后重新创建一个**新提交**，不要 `--no-verify`。
- **push 需要强推**：停止并让用户明确确认，不在本 skill 内自动处理。
- **当前是 `main` / `master` 且上下文看起来像高风险操作**：先提醒用户这是直接推主分支的操作，再继续执行。

## 输出要求

执行完成后，回复应简短包含：

1. 提交是否成功
2. push 是否成功
3. 提交 hash 与中文标题
4. 是否还有未提交改动

如果中途停止，要明确说出停止在哪一步，以及原因。
